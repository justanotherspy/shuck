## The two modes (compatibility contract)

shuck has two modes. They work independently, can be used together, and the
second must never break the first. Every v2 PR is bound by this contract.

1. **Portable mode (default — unchanged).** The `shuck` CLI / MCP server with
   a GitHub token. Pull-based, zero infrastructure. This remains the default
   way to use shuck: every existing subcommand, MCP tool, flag, cache
   behaviour, exit-code contract, the versioned `--json` schema, and the
   Claude Code plugin keep working exactly as today — no backend deployed, no
   channel configured, nothing new to install.
2. **Self-hosted mode (strictly opt-in).** The v2 event router: GitHub App →
   webhook ingest → SQS → workers → gateway → channel shim → Claude Code
   session. Push-based. It is only active when an operator deploys the
   backend **and** a user explicitly installs/configures the channel shim
   with a minted token. No shuck component may auto-enable it.

Rules for every v2 change:

- **No behavioural side effects on portable mode.** Existing CLI/MCP
  behaviour, flags, output, exit codes, and the `--json` schema change only
  additively, never as a by-product of backend work.
- **Channels only on explicit opt-in.** Shim installed + token configured +
  `shuck_subscribe` called. Nothing subscribes a session implicitly, and with
  no shim installed there are no connection attempts, warnings, or behaviour
  differences.
- **The portable binary stays lean.** Backend components are separate
  binaries under `cmd/` (single Go module, but the root `shuck` binary never
  imports backend packages, so AWS SDK et al. are never linked into it).
- **The modes compose.** A channel-subscribed session can still use the
  pull-based MCP tools; channels-blocked environments keep the pull path as
  the fallback.

Shared packages (`internal/distil`, `internal/logs`, `internal/model`, …) are
consumed by both modes; changes there must keep the existing CLI/MCP golden
outputs byte-identical unless the output is *meant* to change.

## JUS-86 implementation plan — webhook ingest

*Status: implemented as planned (same PR). One deviation worth noting: the
first active-fuzz pass on `FuzzIngestFilter` found the filter enqueueing
invalid envelopes for `workflow_run` payloads with a zero run id or empty
`pull_requests[].number`; the filter now drops both, and the minimized
crasher is committed as a regression seed under
`internal/ingest/testdata/fuzz/`.*

Goal: the stateless ingest component — verify, dedupe, filter, enqueue — as a
pure Go core with two thin entrypoints (plain HTTP server and Lambda), plus a
new `shuck-ingest` binary. GitHub App registration is a documented manual
operator step; Terraform wiring ships with JUS-92.

### Package layout

```
internal/ingest/          pure handler core (no AWS imports)
  verify.go               HMAC SHA-256 of X-Hub-Signature-256, constant-time
  filter.go               event-type + action filter → envelope kind or drop
  envelope.go             the versioned SQS envelope type + JSON encoding
  handler.go              http.Handler: verify → dedupe → filter → pre-filter → enqueue
  fuzz_test.go            fuzz the signature check + payload → envelope parse
internal/ingest/awsx/     thin AWS adapters (DynamoDB deduper, SQS enqueuer)
cmd/shuck-ingest/         new binary: HTTP server + Lambda entrypoints
```

The root `shuck` binary does not import any of these packages, so portable
builds stay free of AWS dependencies (module `go.mod` grows; the shipped
binary does not).

### Core contracts

- **Verify.** `X-Hub-Signature-256` HMAC SHA-256 over the raw body with the
  webhook secret, compared with `hmac.Equal`. Unsigned or mismatched → 401,
  body never parsed. Test vectors from the GitHub docs.
- **Dedupe.** `Deduper` interface: conditional put of the `X-GitHub-Delivery`
  GUID with a short TTL; a repeat delivery is dropped with a 200 (GitHub must
  not retry). DynamoDB implementation in `awsx`; in-memory implementation for
  tests and the dev server.
- **Filter.** Table-driven: `workflow_run.completed` with
  `conclusion ∈ {failure, cancelled}` → `ci_failure`;
  `pull_request.closed` → `pr_closed`; everything else dropped before the
  queue. JUS-91 later adds the review kinds here — the table is the extension
  point.
- **Subscription pre-filter.** `SubscriptionChecker` interface
  (`HasSubscriber(repo, pr)`); until the JUS-88 table schema exists the
  default implementation is allow-all, so the pre-filter is a no-op that can
  be tightened without touching the handler.
- **Envelope.** Slim, versioned, never the raw payload:
  `{schema: 1, delivery_id, kind, repo, pr, run_id, head_sha, installation_id}`
  (`pr_closed` omits the run fields). This is the contract JUS-87 consumes —
  it lives in `internal/ingest` and workers import it from there.
- **Entrypoints.** `cmd/shuck-ingest` serves the same handler either as a
  plain HTTP server (`/webhook`, plus `/healthz`) or as a Lambda function
  URL handler (auto-detected via `AWS_LAMBDA_RUNTIME_API`, overridable by
  flag). Config via env: webhook secret (or its Secrets Manager ARN), queue
  URL, dedupe table.
- **Observability.** Structured logs (`log/slog`) + counters: received /
  verified / deduped / filtered / enqueued.

### Testing

- Unit tests for verify (GitHub test vectors, tampered signature), filter
  (table), dedupe (repeat GUID processes once), and the handler wired with
  in-memory fakes — both entrypoints exercise the same suite.
- Fuzz targets per repo convention (`FuzzIngestVerify`,
  `FuzzIngestEnvelope`): signature check fails closed; payload → envelope
  round-trips without panics.
- Coverage stays ≥ 80% (`make cover-check`); the core is pure so this is
  cheap.

## JUS-88 implementation plan — gateway

*Status: implemented as planned (same PR). Deviations worth noting: the
buffer table's sort key is a prefixed string (`s#<seq %020d>`), not the bare
numeric `sk=seq` sketched in the ticket, so the seq counter, event_id dedupe
markers, and presence rows share the subscriber's partition — one table
instead of three auxiliary ones. The "optional" final informational event on
`pr_closed` is always sent (it costs nothing: it is buffered like any other
event, then subscriptions are removed).*

*JUS-92 addendum: this resident server is now one of two gateway shapes —
the serverless deployment target runs the same stores and protocol as API
Gateway WebSocket Lambda handlers (`internal/gateway/serverless`), with a
small additive protocol superset. See § JUS-92.*

Goal: the persistent event-delivery service — WebSocket termination,
per-user token auth, subscriptions, and the write-then-push event buffer —
as a pure Go core with DynamoDB adapters and a long-lived server binary.
Single replica in v1; the connection registry is behind an interface so HA
(JUS-95) slots in without touching the core.

### Package layout

```
internal/gateway/         pure core (no AWS imports; coder/websocket + stdlib)
  deliver.go              the versioned worker → gateway deliver contract
  protocol.go             WS wire frames, close codes, token hashing
  stores.go               TokenStore / SubscriptionStore / EventBuffer /
                          PresenceStore interfaces + domain types
  registry.go             live-conn lookup (interface = the JUS-95 HA seam)
  conn.go, hub.go         connection state machine + handshake/fan-out/drain
  deliverhttp.go          POST /internal/deliver (shared-secret header)
  sweep.go                grace-window sweep of disconnected subscribers
internal/gateway/awsx/    DynamoDB adapters (the only gateway pkg with AWS)
cmd/shuck-gateway/        the server binary: env config, SIGTERM drain
```

### Core contracts

- **Deliver.** `POST /internal/deliver` carries
  `{schema: 1, event_id, kind, repo, pr, summary, author?}` — the contract
  the JUS-87 worker produces (import `gateway.DeliverRequest`). Kinds:
  `ci_failure`, `pr_closed`, `review_comment`, `review` (the review kinds
  arrive with JUS-91; the enum exists now because suppression is
  kind-scoped). Auth is the `X-Shuck-Deliver-Secret` header, compared in
  constant time against one or two configured values (rotation), before the
  body is read — a rejected call can never buffer anything.
- **Wire protocol (shim ↔ gateway)** exactly as the JUS-84 epic:
  `hello {token, session_id, last_event_id?}` / `subscribe` / `unsubscribe`
  / `ack {id}` inbound, `event {id, seq, repo, pr, kind, summary}` outbound.
  Close codes: 4401 unauthorized, 4409 replaced (newest wins), 1001 drain.
- **Identity.** `user_id#session_id` everywhere; `user_id` is resolved from
  `sha256(token)` at hello, so a foreign session id yields an empty
  partition, and self-authored suppression is a pure compare at fan-out
  (numeric GitHub user ID only, review kinds only — never `ci_failure`).
- **Write-then-push.** The per-connection writer goroutine *pulls* from the
  buffer table behind a seq cursor; deliveries persist the row first, then
  nudge the writer. Replay-then-live ordering and crash recovery are
  structural: anything persisted is picked up by the next drain, anything
  not persisted was answered 5xx so the worker retries into the event_id
  dedupe (a marker row written transactionally with the event row).
- **Sweep.** Presence rows (durable `last_seen` / `disconnected_at`) feed a
  periodic sweep that removes subscriptions + buffer of subscribers
  disconnected longer than the grace window (default 24h); live connections
  are never swept, whatever their rows claim after a crash.

### Table schemas (the JUS-92 Terraform contract)

| Table | Keys | Notes |
| --- | --- | --- |
| tokens | `pk` (S) = hex sha256 of token | attrs `github_user_id` (N), `github_login` (S), `repo_allowlist` (L, reserved), `created` (N); revocation = row delete |
| subscriptions | `pk` (S) = `owner/name#pr`, `sk` (S) = `user_id#session_id` | GSI `subscriber-index` (hash `sk`, range `pk`, KEYS_ONLY); attr `created` (N) |
| buffer | `pk` (S) = `user_id#session_id`, `sk` (S) discriminated | `s#<seq %020d>` event rows, `c` seq counter, `e#<event_id>` dedupe markers, `p` presence; TTL on `expires`. JUS-92 adds `w` connection-registry rows (plus `conn#<id>` reverse partitions) for the serverless gateway — same table, same TTL discipline |

## JUS-87 implementation plan — CI-failure worker

*Status: implemented as planned. Deviations worth noting: secrets are
env-injected (deliver secret, App private key PEM or file path) rather than
read from Secrets Manager — the same convention the ingest and gateway
binaries established, with JUS-92/93 owning the injection wiring. The
workflow YAML fetch sketched in the ticket is deliberately absent: `distil`
recovers step commands from the logs themselves (a v1 design rule), so the
worker never reads workflow files. The raw-log S3 archive is best-effort —
an archive failure logs and counts but never fails the envelope, since the
delivered summary is the product.*

Goal: the queue consumer between ingest and gateway — mint an App
installation token, fetch the failed run's jobs and logs, distil, cap, and
deliver — as a pure Go core with AWS adapters and a binary that runs the
same core as a container poll loop or an SQS-event Lambda.

### Package layout

```
internal/worker/          pure core (no AWS imports)
  worker.go               Processor: envelope → token → fetch → distil →
                          cap → deliver; TokenSource / RunFetcher /
                          LogStore / Deliverer interfaces
  apptoken.go             AppTokenSource: RS256 App JWT (golang-jwt/jwt) →
                          installation token, cached per installation
  fetch.go                GHFetcher: production RunFetcher over internal/gh
  deliver.go              HTTPDeliverer: POST /internal/deliver + retry
  metrics.go              atomic counters (latency, truncation, quota gauge)
internal/worker/awsx/     thin AWS adapters (SQS consumer, Lambda SQS
                          entrypoint, S3 raw-log store)
cmd/shuck-worker/         new binary: poll loop + Lambda entrypoints
```

### Core contracts

- **Pipeline.** `ci_failure` envelopes: require `installation_id`, mint a
  token, `gh.RunReport` the run's failed + cancelled jobs (capped at
  `MaxJobs`, default 20, against the queue visibility timeout),
  `gh.JobLog` each (a job whose log is gone degrades to a "logs
  unavailable" summary line, never an error), `distil.CIFailure` per job,
  join, cap, deliver. `pr_closed` envelopes skip fetch and token entirely
  and deliver straight through — that is what removes the PR's
  subscriptions (JUS-88).
- **Summary cap, enforced in distil.** `distil.CapSummary(summary, limit,
  note)` — additive, byte-budgeted (default `distil.DefaultSummaryLimit`,
  16 KiB, `SHUCK_SUMMARY_LIMIT` to tune), keeps whole lines from the start
  (the header and first failing steps carry the first error), cuts at
  UTF-8 rune boundaries, and appends the worker's note. When the raw logs
  were archived the note carries the `s3://…/raw/<repo>/<run_id>/`
  pointer, so a reading agent knows where the whole thing lives; the
  existing `CIFailure` path and its goldens are untouched.
- **App tokens.** RS256 App JWT (iat backdated 1m, exp 9m, PKCS#1 or
  PKCS#8 keys) exchanged at `/app/installations/{id}/access_tokens`;
  the ~1h installation token is cached per installation and refreshed
  within a 5-minute margin of expiry. No PATs anywhere.
- **Idempotency.** The deliver `event_id` is the envelope's delivery GUID
  in every path, so queue redeliveries and deliver retries land in the
  gateway's `e#<event_id>` dedupe markers instead of double-notifying.
  The worker keeps no dedupe state of its own.
- **Retry split.** In-process: 5xx/transport deliver failures retry up to
  3 times with doubling backoff (a 4xx is a contract bug — never
  retried). Cross-process: any envelope error leaves the message on the
  queue (poll mode) or reports a batch-item failure (Lambda); persistent
  failures reach the DLQ via the queue's redrive policy — the worker
  never drops work itself.
- **Raw-log archive.** `raw/{repo}/{run_id}/{job_id}.log`, opt-in via
  `SHUCK_RAW_LOG_BUCKET`; retention is a bucket lifecycle rule (24h
  default) provisioned with JUS-92 — retention config never lives in
  worker code.
- **Observability.** Structured logs + atomic counters: fetch/parse
  latency sums, truncation rate, token mints vs cache hits, deliver
  retries, and a `rate_remaining` gauge — every installation shares one
  App REST quota, so that gauge sinks before all users throttle at once.

### Entrypoints

`cmd/shuck-worker` auto-detects Lambda (`AWS_LAMBDA_RUNTIME_API`), else
long-polls SQS with a `/healthz` listener and a periodic metrics snapshot;
SIGTERM stops the loop cleanly. **The Lambda event source mapping must
enable `ReportBatchItemFailures`** (JUS-92): the handler always returns a
partial-batch response, and without that flag Lambda would delete failed
records with the rest of the batch. Recommended queue visibility timeout
≥ 5m — a many-job run means one jobs listing plus N log downloads.

### Testing

  interfaces (pr_closed never fetches, event_id pinned to the delivery

- Coverage stays ≥ 80% (`make cover-check`) for both new internal

- Queue/DLQ/bucket/lifecycle provisioning and secret injection (JUS-92/93),

## JUS-89 implementation plan — channel shim

*Status: implemented as planned. The spike result first: the identity
assumption holds —* `CLAUDE_CODE_SESSION_ID` *is present in the stdio MCP
server environment (verified empirically on Claude Code 2.1.201 by
inspecting a live stdio MCP server's process environment; it equals the
session UUID that `--resume` preserves). The ephemeral-UUID fallback stays
as a safety net. Deviations worth noting: the ticket's "await ack" for the
subscribe tools is impossible against the frozen JUS-88 protocol — the
gateway sends no reply frame to subscribe/unsubscribe (a store failure
closes the connection instead) — so the tools return success once the frame
is written on an authenticated connection. The event-id dedupe the gateway's
replay path relies on is an in-memory insertion-ordered set capped at 1024
ids; duplicates skip the notification but are still acked, which is what
drains replayed buffer rows. The liveness watchdog uses Bun's non-standard
client `ws.ping()`/`pong` event (verified working on Bun 1.3.11) because the
standard WebSocket API auto-pongs the gateway's heartbeat invisibly; it
degrades to a no-op if the extension disappears.*

*JUS-92 addendum: the shim now speaks a small superset of this protocol so
one shim serves both gateway shapes — in-band `unauthorized`/`replaced`
control frames (the serverless gateway cannot send close codes) and a
5-minute application-level `ping` keepalive, both inert against the
resident gateway. See § JUS-92.*

Goal: the user-facing opt-in switch for self-hosted mode — a stdio MCP
server (TypeScript/Bun, `@modelcontextprotocol/sdk` as the only runtime
dependency) that Claude Code spawns per session, packaged as a Claude Code
plugin **separate from** `plugins/shuck`, modelled on the official fakechat
channel example.

### Layout

```
.claude-plugin/marketplace.json   the repo is now a plugin marketplace
                                  (both plugins; install shuck-channel@shuck)
plugins/shuck-channel/
  .claude-plugin/plugin.json      manifest (validated in CI alongside shuck's)
  .mcp.json                       bun run --cwd ${CLAUDE_PLUGIN_ROOT} start
  server.ts                       MCP entry: capabilities {tools:{},
                                  experimental:{'claude/channel':{}}},
                                  instructions (event kinds + the worker's
                                  s3 truncation marker), tools, notification→ack
  gateway-client.ts               the WS client (hello/backoff/dedupe/ack)
  config.ts, pr-url.ts            zero-noise config resolution; PR URL parsing
  *.test.ts                       bun test vs. an in-process fake gateway +
                                  a stdio smoke test via the SDK client
```

### Core contracts

- **Wire protocol.** Mirrors `internal/gateway/protocol.go` frame-for-frame:
  `hello {token, session_id, last_event_id?}` first, `subscribe` /
  `unsubscribe {repo, pr}`, `ack {id}` out; `event` in. Close-code policy:
  4401 and 4409 stop the client permanently (a bad token or a newer
  connection for the same session); 1001 and everything else reconnect with
  full-jitter exponential backoff (cap 30s, attempt counter reset after 10s
  of stable connection). `last_event_id` is in-memory only — no local state
  beyond the token config; a fresh spawn replays the gateway buffer and
  dedupes.
- **Strict opt-in, zero noise.** Config is `SHUCK_CHANNEL_GATEWAY_URL` +
  `SHUCK_CHANNEL_TOKEN` (or `~/.config/shuck/channel.json`), both required;
  unconfigured ⇒ no sockets, no timers, no stderr — the smoke test asserts
  an unconfigured shim writes zero bytes to stderr. Channels are additionally
  per-session opt-in via `--channels` on the Claude Code side. The shim
  never touches the existing plugin, root `.mcp.json`, or the skill.
- **Events.** `notifications/claude/channel` with `{content: summary,
  meta: {repo, pr, event, event_id}}`, acked only after the notification
  resolves — a failed delivery is left unacked for the next replay.
- **Identity.** `session_id = CLAUDE_CODE_SESSION_ID`, ephemeral UUID + one
  stderr warning as fallback (only when configured).

## JUS-90 implementation plan — token portal

*Status: implemented as planned. Deviations worth noting: GitHub Apps do
not support PKCE, so the GitHub leg's CSRF defense is the session-bound
`state` parameter alone (32 random bytes, HMAC-cookie-bound, constant-time
compared) — the ticket's "PKCE if supported" resolves to "not supported".
The ticket's "open WS connections on the old token are closed" acceptance
is met at the protocol's granularity: revocation is a row delete and the
gateway token lookup is a consistent read, so the old token dies at its
next hello (reconnect); the frozen JUS-88 protocol has no mid-connection
re-auth, and the portal's UX copy says exactly that. `last_used` is a new
additive attribute on the token row, stamped best-effort by the gateway
after a successful hello (`gateway.TokenToucher`; `attribute_exists(pk)`
guard so a touch racing a revocation cannot resurrect the row) — the
frozen row shape the portal writes is unchanged. Fuzzing found real cookie
malleability (non-canonical base64 trailing bits decoding to the same MAC);
the session codec now decodes strictly and the crashers are regression
seeds.*

Goal: the self-service token portal — optional generic OIDC gate → GitHub
App user-authorization → org-membership / account-ownership validation →
mint-shown-once into the JUS-88 token table — as a pure Go core with a
DynamoDB adapter and a small server binary, plus the daily re-validation
sweep.

### Core contracts

- **Token table shape is the frozen JUS-88 contract** — the portal writes
  `pk` (S) = hex sha256 of the raw token, `github_user_id` (N),
  `github_login` (S), `repo_allowlist` (L, written empty, reserved, never
  enforced in v1), `created` (N). Revocation = row delete; the gateway
  rejects the token at its next hello (4401) and the JUS-89 shim stops
  permanently with a "fix the token and restart" error. `last_used` (N) is
  additive: gateway-stamped on hello, portal-displayed.
- **Identity**: users are keyed by the immutable numeric GitHub user ID;
  the login is display/probe material. OIDC (when enabled) only gates the
  UI — its subject is audit-logged, never an identity key.
- **Token hygiene**: `shk_` + base64url(32 random bytes), shown exactly
  once in the POST /token response body (no redirects, `no-store`, never
  in a URL or slog arg — audit lines carry the hash). Regenerate =
  `ByUser` + one `TransactWriteItems` deleting every prior row and putting
  the new one; a crashed half-regenerate self-heals on the next one.
- **Validation modes** (env XOR): `SHUCK_GITHUB_ORG` checks org membership
  with a members:read installation token (App JWT via the shared
  `worker.AppTokenSource`); `SHUCK_GITHUB_ACCOUNT_ID` compares numeric IDs
  locally for personal installs. A validation *error* is always "unknown":
  the handler answers 502 (never a mint), the sweep skips (never a
  revoke) — the same soft-degradation rule as the rest of shuck.
- **Sweep**: daily (`SHUCK_SWEEP_INTERVAL`) scan of all rows; definitive
  non-members are deleted and audited (`token_revoked_sweep`). GitHub org
  membership is the access-control plane — offboarding included.
- **Session security**: single HMAC-signed cookie (≥32-byte secret, 1h
  TTL), strict-base64 fail-closed decode, session rotated on privilege
  change, CSRF form token on the mint POST, `state`/`nonce` bound to the
  cookie and compared constant-time, `Cache-Control: no-store` +
  `Referrer-Policy: no-referrer` everywhere.

### Core contracts

- **One event per human action.** GitHub fires `pull_request_review
  .submitted` for *every* comment action (a lone comment or thread reply
  creates an implicit review: state `commented`, empty body, one comment)
  alongside `pull_request_review_comment.created`; a bulk review fires
  1 + N events. Both worker arms apply the same API-derived rule —
  `standaloneWrapper(state, body, comments)` = commented ∧ empty body ∧
  ≤ 1 comment — order-independently: a standalone wrapper is delivered by
  the comment arm (with thread and file context) and skipped by the
  review arm; anything else is one coherent review event (verdict +
  body + every comment, hunk-only) and its member comment events are
  skipped.
- **Author identity.** Deliver payloads for review kinds always carry
  `author {github_user_id, login}` from the webhook payload (envelope
  validation requires `author_id > 0`; the filter fails closed without
  it), so the gateway's per-subscriber self-authored suppression always
  has its key. The gateway is untouched.
- **Bot guard ≠ self-suppression.** `SHUCK_IGNORE_AUTHORS` (numeric IDs
  and/or logins) is a global drop in the worker, before any token mint —
  an agent commenting AS the App must not trigger anyone. Agents
  commenting as a user are handled per-subscriber by the gateway.
- **Idempotency unchanged.** `event_id` is the envelope's delivery GUID
  in every path; the worker keeps no dedupe state.
- **Degradation.** Object deleted before fetch ⇒ `ErrGone` ⇒ envelope
  consumed, counted, never delivered and never DLQ'd. Parent review
  deleted ⇒ treated as standalone (the review arm sees the same 404 and
  skips, so single-delivery holds). Thread or file-content fetch failure
  ⇒ degrade (bare comment / hunk-only). An unreadable (non-404) parent
  fails the fetch for redelivery — both arms must derive the same
  standalone verdict.
- **Summary cap.** `distil.CapSummary` with the existing
  `SHUCK_SUMMARY_LIMIT`; review summaries use the plain
  `[summary truncated]` note (no raw-log archive applies).

### The protocol superset (shim ↔ serverless gateway)

API Gateway cannot send application close codes, never shows Lambda a
protocol ping, and hard-caps connections at 2h (idle timeout 10m). One
shim (JUS-89) speaks to both gateway flavors via three additions, all
inert against the resident gateway:

- The 4401/4409 verdicts arrive as in-band `{"type":"unauthorized"}` /
  `{"type":"replaced"}` control frames posted before the connection is
  dropped; the shim stops permanently on either, exactly as for the close
  codes.
- The shim sends an application-level `{"type":"ping"}` every 5 minutes:
  it resets API Gateway's idle timer and doubles as the durable presence
  touch the grace-window sweep reads (the serverless gateway has no
  resident process to heartbeat). The resident hub ignores it.
- The 2h cap makes reconnects routine; the existing backoff + buffered
  replay absorbs them. The serverless hello replays the *whole* unacked
  buffer (acks delete rows, so that set is exact) rather than trusting the
  `last_event_id` cursor — with concurrent per-invocation delivers a
  cursor could skip a not-yet-pushed event, and re-sends are free (the
  shim dedupes by event id). Deliver pushes likewise drain everything
  unacked in seq order.

## JUS-96 implementation plan — backend observability

*Status: implemented. Surfaced by the JUS-94 docs pass: the backend had
counters but no way to scrape or alarm on them. Opt-in like everything in v2 —
off by default, no change to portable mode, no new fixed cost.*

Three layers, each independently gated:

- **`internal/promexpo`** — a dependency-free Prometheus text-exposition
  writer (`Write`, `Handler`, `Serve`) and the `SHUCK_METRICS_ADDR` convention.
  Hand-rolled rather than pulling in `client_golang`, matching the repo's
  small-client style (GraphQL/OCI) and keeping the dependency surface off the
  backend binaries. Imports only stdlib, so it never risks the portable
  import-graph guard (verified: `go list -deps .` still clean; promexpo is not
  linked by the root binary).
- **`Snapshot() []promexpo.Sample`** on each backend `Metrics` type
  (gateway/worker/ingest), namespaced `shuck_<component>_<field>` with `_total`
  on counters. The portal had *no* Metrics type — added one (`internal/portal/
  metrics.go`, nil-safe mutators) and wired it at the mint/regenerate/refuse
  sites and the sweep (revoked/unknown/passes), so the counters the RUNBOOK
  always claimed (portal minted/revoked, sweep "unknown") now actually exist.
- **Wiring**: each resident binary (gateway server, worker poll loop, ingest
  server, portal server) starts a dedicated metrics listener via
  `promexpo.Serve` when `SHUCK_METRICS_ADDR` is set — a separate port from the
  app, so `/metrics` is never routed by the app ingress. Lambda modes don't
  serve it (CloudWatch is their path).

Deployment targets:

- **Helm (JUS-93 target)**: `observability.enabled` injects
  `SHUCK_METRICS_ADDR`, adds the `metrics` container + Service port, and — when
  `networkPolicy.enabled` — opens that port (worker's egress-only policy gains
  an ingress rule; ingest/portal ingress blocks now emit when either a peer
  list *or* observability is set). Optional `ServiceMonitor`
  (gateway/portal/ingest) and `PodMonitor` (all pods, incl. the Service-less
  worker) templates, each with interval/timeout/labels/relabelings knobs. The
  combo CI values enable the core wiring (kind applies it; the Operator CRDs
  aren't present there, so the monitors are render-checked separately by a
  third `helm template` in `make helm-check`).
- **Terraform (JUS-92 target)**: `/metrics` has no home in Lambda, so
  CloudWatch is the equivalent. A `var.observability` object gates a DLQ-depth
  alarm (the one the README used to tell operators to build by hand),
  per-Lambda `Errors` alarms, a gateway WebSocket `ExecutionError` alarm, a
  stack dashboard, and Active X-Ray tracing (a `dynamic "tracing_config"` block
  on every function + the X-Ray write policy on each role). `alarm_actions`
  routes notifications. All off by default — idle cost stays ~$0.
